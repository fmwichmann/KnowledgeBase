## Kontext

Der Telegram Bot benötigt eine Laufzeitumgebung, die Nachrichten empfängt und verarbeitet. Für die mittelfristige Anforderung (REQ-005: Arbeitsanweisungen) sollen komplexe, mehrstufige Workflows zuverlässig ausgeführt werden können.

## Alternativen

**python-telegram-bot mit Polling:** Kein Flask, kein ngrok, ein Prozess. Einfachster Einstieg, aber keine native Workflow-Orchestrierung für REQ-005.

**Flask + Webhooks ohne Temporal:** Synchrone Verarbeitung, ein Prozess, kein Docker. Für Q&A ausreichend, für komplexe Arbeitsanweisungen nicht geeignet.

**Flask + Temporal + Webhooks (gewählt):** Webhook-Empfang via Flask, Workflow-Orchestrierung via Temporal. Höherer Initialaufwand (Docker, Worker-Prozess), aber direkt bereit für REQ-005.

## Entscheidung

Flask empfängt Telegram-Nachrichten per Webhook. Temporal übernimmt die Workflow-Orchestrierung. Lokal wird ngrok als temporäre öffentliche URL genutzt; auf dem Server wird ngrok durch die feste Server-URL ersetzt.

Laufzeitkomponenten:
- `bot.py` — Flask-Webhook-Server (Port 5000)
- `worker.py` — Temporal Worker
- Temporal Server — via Docker

## Konsequenzen

**Positiv:** Direkt bereit für komplexe Arbeitsanweisungen (REQ-005). Zuverlässige Ausführung mit automatischen Retries durch Temporal.

**Negativ:** Drei Prozesse müssen laufen (Flask, Worker, Temporal via Docker). Ngrok-URL ändert sich bei jedem Neustart — Telegram-Webhook muss neu gesetzt werden.

**Neutral:** Auf dem Server entfällt ngrok; Temporal läuft als Dienst statt via Docker-Dev-Mode.

## Kontext

ChromaDB benötigt ein Embedding-Modell, das Text in Vektoren umwandelt. Da Dokumente auf Deutsch verfasst sind, aber Flexibilität für spätere Spracherweiterungen erhalten bleiben soll, wird ein multilinguales Modell bevorzugt. Die Implementierung soll lokal erfolgen, ohne Cloud-Abhängigkeit.

## Alternativen

**multilingual-e5-small:** Geringerer Ressourcenbedarf (~470MB), leicht niedrigere Qualität. Sinnvoll bei sehr begrenzten Ressourcen.

**multilingual-e5-large:** 3–5 Prozentpunkte bessere Retrieval-Qualität auf MTEB, ~3–4GB RAM im Betrieb. Upgrade jederzeit durch Neuindexierung möglich.

**deepset/gbert-large-paraphrase-cosine:** Für rein deutschen Inhalt qualitativ überlegen, keine Mehrsprachigkeit. Schränkt spätere Spracherweiterung ein.

**OpenAI text-embedding-3 / Cohere embed-multilingual:** Höchste Qualität, aber API-basiert — widerspricht REQ-007 (lokale Datenhaltung).

## Entscheidung

**multilingual-e5-base** via sentence-transformers, lokal betrieben. Für eine kleine, gut strukturierte persönliche Wissensbasis mit klarer Fachsprache ist die Qualität ausreichend. Ein Upgrade auf e5-large ist durch Neuindexierung jederzeit möglich.

## Konsequenzen

**Positiv:** Kostenlos, lokal, keine Cloud-Abhängigkeit. Multilingual — unterstützt spätere Spracherweiterung. Direkte ChromaDB-Integration via sentence-transformers.

**Negativ:** Modellwechsel erfordert vollständige Neuindexierung aller Dokumente.

**Neutral:** Disk ~1.1GB, RAM ~1.5–2GB im Betrieb, Latenz pro Chunk ~50–150ms auf CPU.
